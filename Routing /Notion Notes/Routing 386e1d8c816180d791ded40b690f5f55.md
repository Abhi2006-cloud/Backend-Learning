# Routing

---

# 1. Why Routing Exists

In the previous topic, we learned that HTTP methods describe **what action** we want to perform.

Examples:

```
GET    -> Fetch data
POST   -> Create data
PUT    -> Replace data
PATCH  -> Update data
DELETE -> Delete data
```

These methods express the **intent** of the request.

But one question remains:

> "Where should this action be performed?"
> 

This is solved by **Routing**.

---

## Routing = The "Where" of a Request

Think of an HTTP request as:

```
HTTP Method + URL Path
```

Example:

```
GET /users
```

Here:

- GET → What to do?
- /users → Where to do it?

The server combines both pieces of information and decides which code should run.

---

## Simple Analogy

Imagine a hospital.

The receptionist asks two questions:

### Question 1

What do you want?

```
Checkup
Surgery
Emergency
```

This is like the HTTP Method.

---

### Question 2

Which department?

```
Cardiology
Neurology
Orthopedics
```

This is like the Route.

---

Only after knowing BOTH answers can the hospital send you to the correct doctor.

Similarly:

```
GET + /users
```

and

```
POST + /users
```

go to different server logic.

---

# 2. Definition of Routing

Routing is:

> Mapping URL paths and HTTP methods to server-side logic (handlers).
> 

---

Example:

```
GET /users
```

might execute:

```python
getAllUsers()
```

while:

```
POST /users
```

might execute:

```python
createUser()
```

---

# 3. Route Matching

Server checks:

```
1. HTTP Method
2. URL Path
```

Together they form a unique route.

---

Example:

```
GET /books
```

Handler:

```python
getBooks()
```

---

Example:

```
POST /books
```

Handler:

```python
createBook()
```

---

Notice:

```
GET /books
```

and

```
POST /books
```

have the same path.

But they don't conflict because methods are different.

---

# 4. Static Routes

## What Are Static Routes?

Routes with fixed URLs.

Nothing changes.

Example:

```
GET /books
```

```
POST /books
```

---

These URLs always remain:

```
/books
```

No variable parts exist.

---

### Why Called Static?

Because:

```
/books
```

is constant.

It never changes.

---

Examples:

```
GET /users
GET /products
POST /orders
```

All are static routes.

---

# 5. Dynamic Routes

## Problem

Suppose we need information about a specific user.

Static route:

```
GET /users
```

returns all users.

How do we fetch only one user?

Example:

```
User ID = 123
```

---

## Solution

Dynamic Routes

```
GET /users/123
```

---

The number changes:

```
/users/123
/users/456
/users/789
```

Hence dynamic.

---

# 6. Route Parameters (Path Parameters)

Frameworks define dynamic parts using:

```
:id
```

Example:

```
/users/:id
```

Meaning:

> Match any value here and store it in a variable called id.
> 

---

Example Request

```
GET /users/123
```

Framework extracts:

```json
{
  "id": "123"
}
```

---

## Route Definition

Example:

```jsx
GET /users/:id
```

Meaning:

```
/users/anything
```

matches this route.

---

Examples:

```
/users/1
/users/10
/users/999
```

All match.

---

# 7. Why Path Parameters Exist

Path parameters represent:

> The identity of a resource.
> 

---

Example:

```
GET /users/123
```

Human-readable meaning:

```
Fetch user whose ID is 123
```

---

Another example:

```
DELETE /posts/50
```

Meaning:

```
Delete post 50
```

---

Path parameters make APIs readable.

This is one core idea behind REST APIs.

---

# 8. Query Parameters

Another type of parameter:

```
GET /search?query=python
```

---

Everything after:

```
?
```

is called a query string.

---

Example:

```
/search?query=python
```

Query Parameter:

```
query = python
```

---

# Structure

```
?key=value
```

Example:

```
?name=abhi
```

---

Multiple Parameters

```
/search?query=python&page=2
```

Parameters:

```
query = python
page = 2
```

---

# Why Query Parameters Exist?

GET requests generally do not have a request body.

Example:

```
GET /search
```

Suppose user types:

```
python
```

How do we send it?

Query parameters solve this.

---

Example:

```
GET /search?query=python
```

---

# Path Parameter vs Query Parameter

## Path Parameter

Represents identity.

Example:

```
/users/123
```

Meaning:

```
Which user?
```

---

## Query Parameter

Represents filtering or metadata.

Example:

```
/users?page=2
```

Meaning:

```
How should data be returned?
```

---

# Easy Rule

Ask:

### Does it identify a resource?

Use:

```
/users/123
```

(Path Parameter)

---

### Does it modify/filter the response?

Use:

```
?page=2
```

(Query Parameter)

---

# 9. Pagination Using Query Parameters

Imagine:

```
GET /books
```

returns:

```json
10000 books
```

Too large.

---

Instead server sends:

```json
{
  "data": [...20 books...],
  "total": 10000,
  "page": 1,
  "totalPages": 500
}
```

---

To fetch page 2:

```
GET /books?page=2
```

---

To fetch page 3:

```
GET /books?page=3
```

---

# Common Query Parameters

## Pagination

```
?page=2
```

---

## Limit

```
?limit=20
```

---

## Sorting

```
?sort=price
```

---

## Order

```
?order=asc
```

or

```
?order=desc
```

---

## Search

```
?query=laptop
```

---

Combined:

```
GET /products?page=2&limit=20&sort=price&order=asc
```

# 10. Nested Routes

Nested routes represent relationships between resources.

---

Example:

```
/users/123/posts
```

Meaning:

```
Posts belonging to user 123
```

---

This creates hierarchy.

```
Users
 └── Posts
```

---

# Nested Route Levels

## Level 1

```
/users
```

Returns:

```
All users
```

---

## Level 2

```
/users/123
```

Returns:

```
User 123
```

---

## Level 3

```
/users/123/posts
```

Returns:

```
All posts of User 123
```

---

## Level 4

```
/users/123/posts/456
```

Returns:

```
Post 456 of User 123
```

---

# Why Nested Routes Matter

Without nesting:

```
/posts/456
```

You know the post.

But not the user relationship.

---

With nesting:

```
/users/123/posts/456
```

Meaning becomes explicit.

---

# 11. Route Versioning

One of the most important production concepts.

---

## Problem

Suppose API v1 returns:

```json
{
  "id": 1,
  "name": "Laptop",
  "price": 1000
}
```

Frontend depends on this structure.

---

Later requirement changes.

You want:

```json
{
  "id": 1,
  "title": "Laptop",
  "price": 1000
}
```

Notice:

```
name -> title
```

This is a breaking change.

---

Old frontend will crash.

---

# Solution: Versioning

Version the API.

---

Version 1:

```
/api/v1/products
```

---

Version 2:

```
/api/v2/products
```

---

Now both work:

```
Old clients -> v1
New clients -> v2
```

---

# Benefits

## 1. Backward Compatibility

Old apps continue working.

---

## 2. Safe Changes

Can introduce breaking changes safely.

---

## 3. Gradual Migration

Frontend teams get time to migrate.

---

# API Lifecycle

```
Create V1
      ↓
Create V2
      ↓
Clients migrate
      ↓
Deprecate V1
      ↓
Remove V1
```

---

# 12. Deprecation

Deprecation means:

> API still works but should no longer be used.
> 

Example:

```
/api/v1/products
```

still functions.

But developers are warned:

```
Move to v2
```

---

Eventually:

```
v1 removed
```

---

# 13. Catch-All Routes

## Problem

User requests:

```
/api/v3/products
```

But route doesn't exist.

---

Without handling:

```
Error
Null response
Confusing behavior
```

---

## Solution

Catch-All Route

Example:

```
*
```

or

```
/anything
```

depending on framework.

---

Meaning:

```
If no route matched,
come here.
```

---

Response:

```json
{
  "error": "Route not found"
}
```

Status:

```
404 Not Found
```

---

# Why Catch-All Routes Are Important

Without it:

```
Bad user experience
```

With it:

```
Clear error message
404 response
Easy debugging
```

---

# Complete Routing Flow

When request arrives:

```
GET /users/123/posts?page=2
```

Server processes:

### Step 1

Method:

```
GET
```

---

### Step 2

Route:

```
/users/123/posts
```

---

### Step 3

Extract Path Parameters:

```json
{
  "id": "123"
}
```

---

### Step 4

Extract Query Parameters:

```json
{
  "page": "2"
}
```

---

### Step 5

Find Matching Handler

```python
getUserPosts()
```

---

### Step 6

Execute Business Logic

```
Database Query
Authentication
Validation
```

---

### Step 7

Return Response

```json
{
  "posts": [...]
}
```
